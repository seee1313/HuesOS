# Rust Unsafe Audit

## Scope and method

This audit covers first-party Rust under `crates/`. It excludes vendored C
source because C has no Rust `unsafe` marker and is governed by a separate FFI
boundary. Counts are generated by `tools/audit-safety.py`; they are review
signals, not a proof of soundness.

Initial branch baseline: 228 unsafe blocks, 49 unsafe functions, and 22 unsafe
impls. After the full Clippy/static-mut pass: 185 blocks, 46 functions, 24
impls, and one foreign `static mut`. The additional marker impls and explicit
blocks encode narrow `UnsafeCell`/raw-pointer contracts that were previously
implicit; the broad raw-tree and syscall-wrapper unsafe surface was removed.

## Changes completed

### Fair run queue

The scheduler's custom WAVL implementation used owning raw pointers, parent and
child links, manual rotations, `Box::from_raw`, recursive raw destruction, and
manual `Send`/`Sync`. It was replaced by `BTreeSet<(vruntime, task_id)>`.
Ordering and asymptotic complexity remain O(log n), while allocation ownership,
destruction, and thread-safety derive automatically from the standard `alloc`
collection.

### ACPI parser

ACPI structures are packed. Dereferencing `*const u32/u64` at firmware offsets
was undefined behavior when the address lacked natural alignment. Parsing now
uses `read_unaligned` value copies, checked address arithmetic, minimum table
and entry lengths, bounded root entry count, and early rejection of malformed
cursor movement. No Rust reference points into packed firmware memory.

### Userspace syscall wrappers

The private syscall trampoline accepts only integer register values. Bad pointer
semantics are rejected by the kernel's validated user-copy layer and cannot by
themselves violate Rust memory safety in libcanvas. Therefore `syscall0..5` are
safe private functions around one documented assembly block. Typed wrappers no
longer add meaningless unsafe blocks around every syscall.

Raw-handle-to-RAII conversion remains unsafe because it represents unique
ownership. Channel now provides `read_optional_handle`, centralizing that
conversion and removing it from init application code.

### Doom adapter globals

Multiple `static mut` values were replaced by one private `UnsafeCell` state
container. Its `Sync` contract relies on DoomGeneric's one-thread,
synchronous, non-reentrant callback model. Mutable references are scoped to a
closure and cannot escape. FFI output pointers are checked for null and have
local safety explanations.

### CPU-local storage

The per-CPU array now uses `UnsafeCell` rather than `static mut`. An atomic
fetch-add assigns every element exactly once; static storage pins addresses for
GS_BASE. The unsafe `Sync` contract documents unique CPU ownership.

### Syscall handler publication

The global mutable handler integer became `AtomicUsize`. Initialization uses a
release store before APs are released. Assembly reads the aligned, immutable-
after-init value. This removes a mutable-static data race.

## Runtime lock-rank boundary review

The all-build lock-rank implementation adds seven localized unsafe blocks, one
unsafe function, and three unsafe `Send`/`Sync` implementations. These sites
cover only invariants that safe Rust cannot encode:

- each GS-installed `CpuLocal` owns one permanently assigned rank tracker;
- local interrupts remain disabled while its `UnsafeCell` is accessed;
- ticket-lock acquire/release provides exclusive access to protected `T`;
- context-switch and ring3-entry wrappers validate that no ranked guard is held
  before calling their assembly symbols.

The rank state has fixed capacity and never allocates. Violations use raw
emergency serial output followed by fail-stop, avoiding recursive acquisition
of the ordinary serial or panic-path locks. Host tests cover valid nesting,
inversion, recursive acquisition, and non-LIFO release. The corresponding
safety-budget increase is intentional; no `unwrap`, `expect`, `panic!`, or
`static mut` count increased.

## uACPI-owned MADT lifetime

The handwritten physical-pointer `RSDP → RSDT/XSDT → MADT` walker was removed.
uACPI now owns discovery, mapping, checksum validation, and reference lifetime;
the SMP consumer receives an ordinary bounded byte slice and repeats all entry
length and arithmetic checks without dereferencing raw firmware pointers. This
removed nineteen unsafe blocks and four unsafe functions from the audited
baseline.

## CR0.WP, SMEP, and SMAP boundary

Four new unsafe blocks are confined to privileged control-register updates and
`STAC`/`CLAC`. CPUID gates optional CR4 features, existing register bits are
preserved, and scoped user-copy guards prevent AC state from crossing IRQ or
syscall boundaries. No raw user-pointer site was added; all existing copies now
run inside the audited guard. This is an intentional security-boundary budget
increase.

## uACPI archive enumeration boundary

Two additional unsafe calls reference installed uACPI tables by index and read
the serialized table count. The wrappers return RAII `Table` references, reuse
the existing bounded SDT-length validation, and copy bytes into an immutable
archive during single-threaded early boot. No foreign pointer escapes into
Ring 3; userspace receives only a read-only VMO snapshot.

## Remaining high-priority boundaries

### Allocator internals

Buddy/slab free lists place link metadata inside free memory. Raw pointers are
expected but need private checked pointer types, stronger range/alignment
validation in release builds, double-free detection, and documented lock
ownership.

### Paging and PMM

HHDM/page-table dereference is inherently unsafe. Follow-up work should replace
panic-on-map failures with typed errors, distinguish active vs inactive address
spaces in the API, and encode ownership of intermediate/data frames.

### SMP bootstrap and stacks

AP trampoline data, per-CPU GDT stacks, and context switching require fixed
addresses and raw assembly. Static mutable stack declarations should move to
explicit `UnsafeCell` storage, and every AP boot phase should be represented by
a state transition rather than comments alone.

### FFI

DoomGeneric supplies raw screen and callback pointers. The adapter validates
nulls and lengths where possible. The C engine remains trusted code inside its
own process; a fault is process-contained by HuesOS.

### Fault probes

The fault-probe program intentionally executes invalid memory access and
privileged/undefined instructions. Its unsafe blocks are test payloads, not
production application logic, and are feature-purpose documented.

## Review rules for new unsafe

A new unsafe site must answer all of the following in an adjacent `SAFETY:`
comment:

1. Which pointer/register/instruction invariant is required?
2. Who establishes it?
3. For how long does it remain valid?
4. What prevents aliasing or a data race?
5. What bounds/alignment/initialization checks were performed?
6. Can malformed userspace or firmware input reach the operation?
7. Why can the operation not be expressed through an existing safe wrapper?
8. Which test exercises the boundary and its failure case?

Unsafe added only to silence the compiler or wrap a safe syscall helper is not
accepted.
